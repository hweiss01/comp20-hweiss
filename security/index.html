ScoreCenter Security Report

Introduction

This security assessment tests the 2048-Game Center created by fellow student Daniel Kim for Comp20 assignment 4. This assignment's requirements were that the developer create a web application that maintains a scoreboard of high scores for the players of 2048, such that multiple users can submit and retrieve scores. Users are also able to search for scores by unique username. However, the developer left many security risks in the product that make it inappropriate to release to the public. This assessment attempts to identify and outline some of these vulnerabilities present and show solutions as to how to prevent them.

Methodology

First, I performed "black box" testing, which meant I attempted to hack the application before reading the source code. I used the curl tool in order to try control the data submitted to the application. Then I looked through the source code of the application and found some errors in there that could be exploited.

Issues Found

  1. Cross Site Scripting

    This was found on the /submit.json page

    This is a highly severe issue, as a cross site scripting attack can completely change what website your viewers see upon accessing your application.

    I found this issue by using curl to add a record to the mongodatabase that had a script command as the username.

    curl command here and screen shot

    This can be avoided by combing user input and either just not accepting posts that include script commands, or escaping to the HTML-code equivalents.

  2. CORS without validation

    Both the POST /submit.json and the GET /scores.json APIs are effected directly. However due to the nature of the code, all APIs are effected.

    This is a very big problem

    I found this problem while looking through the source code of the application. The code written uses the wild card "*", which allows any domain to gain access to the application. It is very unwise to open up an application universally like this. Also, this code allows CORS for every API, instead of just the APIs that require CORS to function properly.

    This can be avoided by either validating the response, or adding some form of user authentication, such that only those with the right credentials can access the site
